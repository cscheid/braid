# braid 0.3.0 — features needed before q2 adopts braid

**Audience:** an agent working in the **braid repo** (`~/rooms/room-1/braid`),
not in q2. This doc is self-contained: hand it off as-is.
**Target release:** braid 0.3.0.
**Why:** q2 (a ~1145-issue project) is migrating from beads_rust to braid. The
migration is unblocked *mechanically* (import preserves ids; scale validated
at 460 KB / sub-second), but four quality gaps should land first so the q2
workflow doesn't regress. None of these are architecturally hard.

All four are independent and can be done in any order / in parallel. Follow
braid's own conventions: `braid-core` holds schema + pure logic (no I/O),
`braid` holds CLI/config/cache/sync. Tests: `cargo test --workspace` (120+
tests, no network). Honor TDD — write the failing test first.

File:line references below were captured 2026-06-08 against the then-current
braid `main`; re-confirm before editing.

## Feature 1 — `braid dep tree <id>` (recursive)

**Gap:** braid has `dep list <id>` (one hop, both directions,
`crates/braid/src/...` dep subcommands) and `dep cycles`, but no recursive
tree view. beads has `br dep tree <id>`, used in q2's CLAUDE.md quick-ref and
by humans orienting in an epic.

**Spec:**
- New subcommand `braid dep tree <id> [--json]`.
- Walk dependency edges recursively from `<id>`, rendering an indented tree.
  Default direction should match what's most useful for epics: show
  **children/descendants** (follow incoming `parent-child` and, sensibly, the
  blocking edges). Decide and document the exact edge set; at minimum cover
  `parent-child` so epic → subtask trees render.
- Mark each node with its status (e.g. `[closed]`, `[open]`) like `dep list`
  already does.
- Detect and gracefully break cycles (reuse `dep cycles` logic in
  `braid-core`) — print a `(cycle)` marker rather than looping forever.
- `--json` emits a nested structure for machine consumption.

**Tests (write first):**
- A small skein: epic A with children B, C; C has child D. `dep tree A`
  renders A→B, A→C→D with correct indentation and statuses.
- A cycle (A blocks B, B blocks A) terminates with a cycle marker.
- `--json` shape is stable and documented.

## Feature 2 — `braid create --deps <type>:<id>[,<type>:<id>...]`

**Gap:** beads supports atomic create-and-link:
`br create "..." --deps discovered-from:<parent>`. braid's `create` has no
such flag (`CreateOpts` in `crates/braid/src/ops.rs:141`), forcing a separate
`braid dep add`. q2's "file discovered work as you go" loop uses the one-shot
form heavily.

**Spec:**
- Add a repeatable `--deps <type>:<target-id>` flag (or comma-separated list)
  to `braid create`.
- After the strand is created, add each dependency (`<new-id>` depends on
  `<target-id>` with the given type), reusing the existing `dep add` code
  path so validation/normalization is identical.
- `<type>` accepts the same strings as `dep add --type`
  (`blocks|parent-child|conditional-blocks|waits-for|related|discovered-from|
  replies-to|duplicates|supersedes|caused-by|...`; `DependencyType::from` in
  `crates/braid-core/src/schema.rs:217` already tolerates unknowns).
- Dangling targets are legal in braid (schema README: "Dangling dependency
  targets are legal and never block") — so don't hard-fail if the target
  doesn't exist yet; match `dep add`'s behavior.
- Update `braid agents-info` and `create --help` to show the flag.

**Tests (write first):**
- `create "x" --deps discovered-from:<id>` produces a strand whose
  `dep list` shows the edge with the right type.
- Multiple `--deps` create multiple edges.
- Bad format (`--deps notacolon`) errors clearly.

## Feature 3 — `braid import`: recognize & skip beads tombstones

**Gap:** beads JSONL can contain soft-deleted records with
`"status": "tombstone"` and `delete_reason`/`deleted_at`/`deleted_by` fields.
braid's `convert_status` (`crates/braid/src/import.rs:114`) only special-cases
`completed`→`closed`; `tombstone` falls through to `Status::Other("tombstone")`
(`crates/braid-core/src/schema.rs:127`), which is neither active nor terminal
and shows up as **noise in `braid list`** (confirmed: q2's JSONL has 2 such
records, `bd-1xf5` and `bd-298oe`, and both appear in `braid list`).

**Spec:**
- During `import`, detect tombstone/deleted records and **skip them**
  (do not create a strand). Detection: `status == "tombstone"` OR presence of
  `deleted_at`/`delete_reason`/`deleted_by`. Be conservative — only skip
  records that are unambiguously beads tombstones.
- Report the count: `imported N strands (skipped M tombstones) from <file>`.
- Make sure an *already-imported* strand is not resurrected: importing a file
  where a previously-live id is now a tombstone is an edge case — document the
  chosen behavior (recommended: skip, leaving any prior strand untouched;
  deletion is a separate explicit op).

**Tests (write first):**
- Import a JSONL with one normal + one `tombstone` record → only the normal
  strand exists; skip count == 1.
- Import a record carrying `deleted_at` but `status:"closed"` → decide and
  test (recommended: treat as tombstone/skip, since beads writes both).
- Existing import round-trip / upsert tests still pass (the export format has
  no tombstones, so braid→braid round-trips are unaffected).

## Feature 4 — `braid` agents-info skill installer (beads-AGENTS.md parity)

**Gap:** beads ships `br agents` to install an `AGENTS.md` workflow stub. braid
has `braid agents-info` (prints the version-matched guide) and the README
mentions "how to install a one-paragraph skill that defers to it", but q2 needs
a concrete, idempotent installer so a project's agents pick up braid usage
without hardcoding commands.

**Spec:**
- Provide a command (e.g. `braid agents-info --install <dir>` or a documented
  one-liner) that writes a small skill/instruction file whose body is "run
  `braid agents-info` for the authoritative, version-matched usage guide" plus
  the one-paragraph workflow summary.
- Idempotent: re-running replaces a managed block in place (use delimiter
  markers, like q2's `<!-- BEGIN/END WORKTREE CONTEXT -->` pattern), preserving
  surrounding user content.
- Keep it tool-host-agnostic in content, but the q2 consumer will drop it under
  `.claude/skills/braid/`. Coordinate the exact path with q2 if needed — the
  installer just needs to emit the file; q2 wires it into its skill system.

**Tests (write first):**
- Install into a temp dir → file exists with the managed block.
- Re-install → no duplication; managed block replaced; trailing user content
  preserved.
