# braid-viewer — Tauri v2 desktop app

Native desktop shell (Tauri v2) wrapping the `ui/` React app. Thin Rust
backend: a few `invoke` commands over `braid-config` (project registry +
per-project `ui_config`); the **webview syncs directly** to the automerge sync
server — no CRDT traffic passes through this process. Background:
[docs/viewer.md](../../docs/viewer.md) and the plan at
`claude-notes/plans/2026/06/12/braid-viewer-tauri.md`.

## Build & run — read this first

- **Run/build only via** `cargo xtask viewer-dev` (`cargo tauri dev`) or
  `cargo xtask viewer-build` (`cargo tauri build`). Requires
  `cargo install tauri-cli --version '^2' --locked` and, on Linux, the
  WebKitGTK deps listed in docs/viewer.md.
- **Never `cargo build --release -p braid-viewer` to _launch_ the app.** In
  Tauri v2 a runnable binary needs the `custom-protocol` feature, which only the
  Tauri CLI sets. A plain cargo build produces a **dev-mode** binary that loads
  `http://localhost:5173` (the Vite dev server) → a blank `ERR_CONNECTION_REFUSED`
  window when no dev server is running. Do **not** add a `custom-protocol`
  feature manually (it's CLI-managed since v2.0.0-beta.5). A bare
  `cargo build -p braid-viewer` is a **compile smoke check only** — that's what
  the `viewer.yml` CI job runs.
- Excluded from `default-members` (Tauri can't build on musl), so
  `cargo xtask ci` and plain `cargo build`/`cargo test` skip it. Build or check
  it explicitly with `-p braid-viewer`.

## Secret hygiene (non-negotiable)

The doc id is a **bearer secret**. Never log `UiConfig` / `docUrl`. The viewer
registry (`~/.config/braid/viewer.toml`) holds **folder paths only** — never
secrets. `get_config_cmd` returns the `automerge:`-prefixed URL, never the raw id.

## Debugging

`tauri-plugin-log` writes to stdout, a rotating `braid-viewer.log` in the OS log
dir, and the webview console; the `devtools` feature is enabled so the
**release** exe is inspectable (right-click → Inspect). Startup logs each
window's resolved URL — a `localhost:5173` there means you built a dev-mode
binary (see above). The release exe has no console on Windows
(`windows_subsystem = "windows"`), so that log file is the primary trace.

## Gotchas

- The main window is built **programmatically in `setup`** so we can pin
  `data_directory` (deterministic IndexedDB / offline cache); consequently
  `tauri.conf.json` has `"windows": []`. Don't re-add a config-defined window.
- `gen/schemas/*.json` are **generated by `tauri-build`** from `capabilities/` —
  don't hand-edit; they regenerate on build (commit the result).
- `time` is pinned `=0.3.47` (Rust ≥1.94 coherence vs tauri's blanket `From`
  impls). Removing the pin breaks the build.
- The viewer's cache (automerge-repo **IndexedDB**) is independent of the CLI's
  `samod` cache (`~/.cache/braid`) — separate replicas converging via the sync
  server, by design. Don't try to share them.
- CSP `connect-src` is static-default (`wss://sync.automerge.org`) only.
  Non-default / self-hosted sync servers need runtime CSP injection (deferred;
  `_extra_servers` is already computed in `run()`, ready to wire in).
